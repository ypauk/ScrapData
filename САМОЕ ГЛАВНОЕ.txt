Единый инструмент: ai_workflow.py

https://github.com/settings/copilot/features

01. python ai_workflow.py new <имя_проекта>
02. Заполнить AI_INPUT/ (description.txt, page.html, network.har...)
03. python ai_workflow.py analyze <имя>
04. Сохранить ответ → AI_OUTPUT/analysis.md
05. python ai_workflow.py project <имя>
06. Сохранить ответ → AI_OUTPUT/project_plan.md
07. python ai_workflow.py module scraper <имя>
08. Сохранить код → app/scraper.py
09. python ai_workflow.py module parser <имя>
10. Сохранить код → app/parser.py
11. python -m app.main  (из папки проекта)
12. При ошибке: сохранить traceback в AI_OUTPUT/traceback.txt → python ai_workflow.py debug <имя>
13. python ai_workflow.py docker <имя>
14. Сдать клиенту

После каждого проекта оценивай систему

Не только "код работает", а по чек-листу:

✅ 01_analyze_project.md выбрал правильную стратегию?
✅ 02_generate_project.md спроектировал адекватную архитектуру?
✅ 03_generate_module.md создал хороший код без лишних функций?
✅ Не пришлось менять промпты?
✅ Не пришлось вручную объяснять ИИ, что делать?
✅ Код сразу вписался в твою универсальную структуру?
✅ Docker (если нужен) собрался без проблем?

Если на каком-то типе задач промпты начинают "сыпаться", не исправляй конкретный проект, а улучшай шаблоны. Именно так твоя система станет сильнее с каждым тестом.

Я бы поставил цель пройти 20–30 разных сценариев до первого серьезного поиска заказов. После этого ты увидишь повторяющиеся закономерности и сможешь доработать промпты один раз, а не "лечить" каждый новый проект отдельно.

1. Workflow

После каждого проекта задавать себе вопросы:

На каком этапе ИИ ошибся?
Где не хватило входных данных?
Где пришлось писать дополнительный промпт?
Что пришлось объяснять второй раз?

Если ответ "да" — значит, в workflow не хватает этапа или инструкции.

2. Промпты

Например, если на нескольких сайтах ИИ:

забывает обработать пагинацию;
не сохраняет cookies;
не добавляет retry;
не пишет логирование;

значит, это нужно внести в базовый промпт, а не исправлять каждый раз вручную.

3. Шаблоны

Во время работы наверняка появятся удачные функции.

Например:

wait_for_network_idle()

download_images()

parse_price()

extract_phone()

save_to_excel()

load_cookies()

Если они уже использовались в нескольких проектах — их стоит перенести в общие шаблоны.

4. Memory

Я бы после каждого проекта добавлял небольшую заметку:

Site:
https://...

Technology:
React

Authentication:
Cookies

Pagination:
Infinite Scroll

Problems:
Cloudflare

Solution:
Playwright + API

Через 30–50 проектов получится очень ценная база знаний.

Самое главное — улучшать критерии качества

Это то, что часто упускают.

После каждого этапа система должна сама проверять:

код запускается;
все импорты существуют;
нет неиспользуемых функций;
нет TODO;
нет pass;
нет фиктивной реализации;
есть обработка ошибок;
есть логирование;
есть type hints;
соблюдается единый стиль.

Чем больше таких автоматических проверок, тем меньше времени уйдет на ручную отладку.

Я бы даже ввел понятие "Regression"

Как в обычной разработке.

Например, после завершения проекта ты обнаружил проблему:

GPT не обработал бесконечную прокрутку.

Ты не просто исправляешь код, а:

Добавляешь новый тестовый сайт в коллекцию.
Улучшаешь промпт.
Проверяешь, что предыдущие проекты по-прежнему проходят успешно.

Так у тебя появится собственный набор эталонных задач.

Например:

tests/

001_static_html/
002_shopify/
003_react/
004_api_only/
005_login/
006_cloudflare/
007_infinite_scroll/
008_pdf/
009_ajax/
010_graphql/

Каждый раз после изменения workflow можно прогонять систему на этих проектах. Если какой-то из них начал работать хуже — значит, новое изменение внесло регрессию.
